1. Project Overview

MediCore HMS (Hospital Management System) is a full-featured healthcare management platform delivered across two parallel implementations: a React Native mobile app (powered by Expo) and a Vanilla JS web application. Both platforms share identical data models, the same seed dataset, and equivalent functionality — giving hospital staff a consistent experience whether they access the system from a smartphone or a desktop browser.

The system covers every major hospital workflow: patient registration, OPD queuing, appointments, clinical admissions, nursing notes, laboratory & radiology orders, pharmacy dispensing, billing, inventory management, staff management, reporting, emergency triage, real-time messaging, and role-based access control.

12. Web Navigation & RBAC

The nav.js module runs on every dashboard page. It renders the sidebar dynamically based on the current user's role from sessionStorage, and guards pages against unauthorized access.

nav.js Functions

FunctionDescription
getUserRole()Reads sessionStorage.hms_user, returns role string. Defaults to "Administrator" if missing.
canAccess(item, role)Returns true if item has no role restriction, or if the user's role is in the item's roles array.
renderNav()Builds and injects the full sidebar HTML into .sidebar-nav element. Filters items by role. Adds active class to current page. Renders live badges (appointment count, critical count, unread messages).
guardPage()Checks if the current page URL requires a specific role. If user lacks access, shows a toast and redirects to dashboard.html.

Session Management (Web)

StorageKeyValueLifecycle
sessionStoragehms_user{ name, email, role }Set on login, cleared when browser tab closes
localStoragehms_data_v1Full dataset JSONPersists across sessions. Seeded from SEED on first visit.
localStoragehms_registered_usersArray of user accountsGrows as users register
⚠️ The web platform uses sessionStorage for the user session — closing the browser tab will log the user out. This is by design for security in shared environments. The mobile app uses AsyncStorage which persists across app restarts.

13. Web Data Layer (HMSData)

data.js exposes a global singleton window.HMSData that provides all CRUD operations over localStorage. Every dashboard page uses this API exclusively — no fetch/XHR calls, no backend.

HMSData API

MethodSignatureDescription
getAll()() → ObjectReturns the full dataset object from localStorage
get(key)(key) → Array|ObjectReturns a single collection by key (e.g., 'patients')
set(key, value)(key, value) → valueReplaces an entire collection and saves
add(key, item)(key, item) → itemAppends item to collection array and saves
update(key, id, updates, idField?)→ updatedItem | nullMerges updates into matching item. Default idField is 'id'.
remove(key, id, idField?)→ voidRemoves item with matching id from collection
find(key, id, idField?)→ item | nullReturns first item matching id, or null
nextId(prefix, key)(str, str) → stringAuto-generates next sequential ID. E.g., nextId('P-', 'patients') → 'P-007'
getPatientName(p)(patient) → stringReturns "firstName lastName"
getAge(dob)(dateStr) → numberCalculates age from ISO date string
formatDate(d)(dateStr) → stringReturns locale-formatted date like "Jun 29, 2026"
badgeForStatus(status)(str) → cssClassMaps status string to a CSS badge class (e.g., 'badge-success', 'badge-danger')
avatarUrl(name)(str) → urlReturns a ui-avatars.com URL for the given name
reset()() → dataClears localStorage and reloads from SEED data
// Example usage on any dashboard page const patients = HMSData.get('patients'); const newPatient = { id: HMSData.nextId('P-', 'patients'), firstName: 'Jane', ... }; HMSData.add('patients', newPatient); HMSData.update('appointments', 'A-1001', { status: 'Completed' }); HMSData.remove('staff', 'S-003');

17. Known Limitations & Future Improvements

Current Limitations

#LimitationImpact
1 No real backend / database All data is stored locally (AsyncStorage / localStorage). Data does not sync between devices or users. Each device/browser maintains its own isolated dataset.
2 No real authentication Passwords are stored in plain text in localStorage/AsyncStorage. This is acceptable for a demo/prototype but must be replaced with hashed passwords and server-side auth before production use.
3 Web session ends on tab close sessionStorage is cleared when the browser tab is closed, requiring re-login. Consider using localStorage with an expiry token for persistent web sessions.
4 No real-time updates Data changes made by one user are not reflected for other users. There are no WebSocket or polling mechanisms. Each user sees only their own local state.
5 Storage size limits AsyncStorage and localStorage have limits (~5–10 MB). Large datasets with images or attachments would exceed these limits.
6 Images loaded from Unsplash CDN Dashboard hero, promo, and service card images are loaded from Unsplash URLs. These require an internet connection and may be unavailable offline or if the URLs change.
7 No PDF export implemented The Reports screen mentions exportable PDFs, but no actual export function is implemented in the codebase. This would require a library like jsPDF or a server-side PDF renderer.
8 Dark mode incomplete (App) The dark mode toggle is functional (toggle + persist), but most screens do not have dark-mode-specific styling applied. The preference is saved but visual theming is partial.
9 No input validation on forms Add Patient, Add Appointment, and Register screens do minimal validation. Duplicate emails, malformed dates, or empty required fields are not consistently caught.
10 Web appointment form is non-functional The landing page appointment form shows a toast confirmation but does not actually save the data or send a notification. It is a UI prototype only.

Recommended Future Improvements

  • Backend API — Replace localStorage with a REST API (Node.js/Express + PostgreSQL or MongoDB) to enable multi-user real-time access
  • Authentication — Implement JWT-based auth with bcrypt password hashing and secure HTTP-only cookies
  • Real-time updates — Add WebSocket support (Socket.io) for live emergency alerts, OPD queue updates, and messaging
  • Push notifications — Integrate Expo Notifications for critical patient alerts on mobile
  • PDF export — Implement jsPDF or react-native-pdf for billing invoices and patient discharge summaries
  • Image uploads — Add profile photo upload for patients and doctors using Expo ImagePicker + cloud storage
  • Dark mode — Complete dark mode theming across all screens using the existing Colors.darkBg / Colors.darkCard tokens
  • Input validation — Add Yup/Zod schema validation on all form inputs
  • Audit log — Track all data changes (who changed what and when) for HIPAA compliance
  • Multi-language support — Add i18n for regional hospital deployments
